Git Conventions

Branching Strategies

Main Branches

master/main: The primary branch representing the most stable, production-ready code. Protect this branch to prevent accidental changes.
develop: The primary development branch where feature branches merge. It should represent code that is ready for testing and integration.

Feature Branches

Create a new branch for each feature.
Use clear and descriptive branch names with a standard prefix (e.g. feature/new-login-page). Branch names are always written in lowercase and use a dash instead of a space.
Keep feature branches focused on a single change for easier review and merging.

Hot fix Branches

Create a new branch for each bug fix, or change.
Use clear and descriptive branch names with a standard prefix (e.g. hotfix/database-error). Branch names are always written in lowercase and use a dash instead of a space.
Keep hot fix branches focused on a single change for easier review and merging.

Commits

Write Messages Containing Verbs: Clearly describe your changes, providing context for reviewers. Use the present tense ("Added feature X" instead of "Add feature X").
Keep Commits Atomic: Each commit should represent a small, self-contained change. This improves maintainability and makes it easier to revert if needed.

Pull Requests

Descriptive Titles: Clearly summarize changes in the title, making reviewing easier.
Assign Reviewers: Request reviews from specific teammates to ensure prompt feedback.
Assign Owners: This helps with tracking code changes and who made them.
Review Process: Thoroughly review code changes, provide constructive feedback, and request revisions if necessary. Reviews will be conducted by the lead developer.
Drafts for Early Feedback: If you'd like feedback before a feature is complete, open a draft pull request. This encourages collaboration early in the development process.

Merging

When Approved: The lead developer will notify the pull request creator and merge it if ready. If changes are required, the lead developer will collaborate with the creator to make them.
Managing Merge Conflicts with a Queue: When multiple pull requests have the potential to conflict, the lead developer may establish a merge queue. This ensures changes are merged in a safe and controlled order. The lead developer will communicate the queue and any updates in the communication channel.
Assets

Overview

This guide outlines the systematic approach from the initial concept of an asset through its creation and eventual implementation. Our workflow is designed to maintain high standards of quality and ensure seamless integration into the project environment.

Models

Initial Assessment

Asset Type Identification: We commence by identifying the asset type, which can be Static Mesh, Hard Surface, or Organic.
Animation Requirement: The necessity for animation is assessed next. For assets requiring animation, a low-detail blockout is created to capture the basic form before adding detailed features.

Sculpting and Detailing

Organic Models: For organic types, Blender or ZBrush is employed for detailed sculpting. Subsequently, models sculpted in ZBrush are imported back into Blender, which is used for retopology and UV unwrapping.

Asset Guidelines

Max polycount character: between 15,000 and 20,000 poly’s.
Max polycount environment: 10,000 to 20,000 poly’s.
Max polycount assets: 1,500 to 3,000 poly’s.

Substance Painter Texturing Process

Application and Baking: Textures, including normal maps, are applied and baked. The process involves using generators, filters, and fill layers without specific settings.
Finalization: After adjusting export settings to PBR Metal Rough, textures are re-imported into Blender for a final review.

Photoshop Texturing Process

Guideline Drawing: The UV map is exported to Photoshop for guideline drawing, then brought back to Blender for shading application. This iterative process continues until the desired outcome is achieved and approved.

UV Mapping

Workflows

Character Models: For characters, UV islands are divided over joints to minimize distortion.
Low-detail Models: A maximum of three islands is maintained for simplicity and efficiency.
Automatic UV’s: The 'mark seam' technique is used for complex objects to optimize UV mapping in Blender, aiming to straighten islands/faces as much as possible.

Rigging

Skeleton Construction

Start with the Root: The process begins at the root bone, setting the foundation for the entire skeletal structure.
Build Upwards: From the pelvis, the skeleton is constructed upwards, extending to the neck and head, and then branching out to the limbs. When working on bones that are going to be mirrored for symmetry, a L/R prefix will be given to the bones, followed by the name of the bone in lowercase (e.g. Lupperleg) before mirroring.

Weight Painting

Automatic Weighting: Initial weight distribution is done automatically by linking the skeleton to the mesh using the “Parent” menu of the object in Blender.
Manual Adjustments: Final adjustments are made in weight paint mode, fine-tuning the mesh deformation for accurate and realistic movement.

Finalization

Review and Export: Follows the same export process as Models.

Animations

Setting Up

Determining Needs: Assess if multiple animations are required for the rig. If so, access the dope sheet and switch to the action editor.
Starting New Actions: Initiate by selecting "new action" for each animation required.

Keyframing

Establish Core Poses: Begin by keyframing essential poses, ensuring a solid foundation for the animation.
Refining Motion: Add detail by filling in between key poses, adhering to consistent frame intervals (e.g., 0, 10, 20) for smooth transitions.

Looping and Organic Motion

Looping: For animations intended to loop, align the first and last frames precisely.
Organic Movement: Review and possibly adopt uneven frame steps to achieve more natural motion. Exaggerated poses may be employed for enhanced organic effect.

Finalization

Review and Export: Follows the same export process as Models.

UI/UX Design

Mockup Creation

UI/UX is designed and created using Inkscape.
Aspect Ratio: Mockups for menus and interfaces are designed with a 1080x1920 aspect ratio.
Export Process: Elements like buttons are exported in PNG format using the "selected only" option, adhering to Power of 2x2 resolutions for optimal compatibility.

Unity Naming Conventions

Scenes Objects

Choose names that clearly describe the object's function or appearance. Use Title Case (e.g., Player, Obstacle Spikes, Door Main Entrance).

Scenes

Scene names are written in PascalCase (e.g., MainMenu, ForestLevel)

Folders

Folder names are written in PascalCase (e.g., Scripts, Prefabs, Textures, Audio, Sound Effects)

Animations

Animation names are written in PascalCase and use the prefix of A_ (e.g., A_Jump, A_Walk)

Materials

Material names are written in PascalCase and use a prefix of M_ (e.g., M_Concrete, M_Wood)

Particle Systems

Particle system names are written in PascalCase and use the prefix P_ (e.g., P_Explosion, P_SmokeTrail)

Sounds

Sound names are written in PascalCase and use the prefix S_ (e.g., S_AmbientWind, S_Footsteps)

Skeletal Meshes

Skeletal mesh names are written in PascalCase and use the prefix SK_ (e.g., SK_PlayerCharacter, SK_EnemySkeleton)

Static Meshes

Static mesh names are written in PascalCase and use the prefix SM_ (e.g. SM_Rock, SM_Tree)

Textures

Texture names are written in PascalCase and use the prefix T_ (e.g., T_GrassDiffuse, T_MetalRust)
