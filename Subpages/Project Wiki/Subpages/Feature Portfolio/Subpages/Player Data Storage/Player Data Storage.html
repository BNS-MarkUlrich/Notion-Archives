Description

This document details the design choices for storing player data in our HTML5 game prototype. It outlines the need for persistent data storage to ensure progress is saved between sessions, fostering a more engaging player experience. The document explores various storage solutions, including external servers, cloud platforms (AWS), and local hosting.

Functional Design

To accommodate the short session nature of HTML5 games and player preferences, we wanted to add a mechanism to save player data, including level progress, across sessions. This ensures that when a player quits the game and returns later, their progress is retained, creating a more engaging and seamless gameplay experience.

Technical Design

In designing a persistent storage solution for player data, we explored several avenues, each offering unique advantages and drawbacks.
Our initial research focused on leveraging an external server equipped with a hosted database. This approach held promise, particularly since we identified a free service (https://www.freemysqlhosting.net/) with sufficient storage capacity (5 MB) for our prototype's needs. Each player data entry encompassed a minimal footprint, consisting solely of two integer values: a player identifier and their experience points. Calculations confirmed that the 5 MB storage limit, translating to roughly 5,242,880 bytes, comfortably accommodated 655,360 data entries – a more than adequate capacity for a prototype. However, this service presented a significant hurdle: the inability to upload custom scripts (PHP) for interacting with the database through API calls. This limitation proved to be a deal-breaker.
Amazon Web Services (AWS) emerged as another potential solution, enticing us with its free tier storage option. However, the complexity associated with setting up AWS, compounded by the team's lack of prior experience with the platform, resulted in delays and ultimately prevented its successful integration within the prototype's development timeline.
Locally hosting the database surfaced as a potential option as well. While this approach possessed a certain appeal, it ultimately became impractical due to the separate dedicated hosting solution employed for the game server itself.
Given the aforementioned considerations, we ultimately opted to leverage Unity's PlayerPrefs functionality for storing player data. This approach provides a straightforward solution for persisting player data across sessions, aligning perfectly with the prototype's development stage. The only major downside this solution has is that it is unsafe and not cheater proof due to the player data being stored on each client own machine so they have the ability to change it. We are aware of this issue but still chose this solution due to that no sensitive data will be stored and this being a prototype and the fact that it wont be publicly released.
In conclusion we have tried a lot of different approaches for saving the player with each solution having their up sides and down sides. We went for saving the data locally due to big learning curve of back end development and the complexity of correctly saving player data.
