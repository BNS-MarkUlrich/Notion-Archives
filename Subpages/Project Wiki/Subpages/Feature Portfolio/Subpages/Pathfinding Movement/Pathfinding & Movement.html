Description

This document delves into the functional and technical design of the pathfinding and movement in our game. It outlines the mechanics of movement, pathfinding, and navigation of the game's agents in the virtual world.
ALT

Functional Design

Movement

In order to move agents in our game, we need a system which can interpolate two vector points in the world and create a path between them.
Generally speaking, agents will calculate a path from where they are to where they want to go, it will then proceed to calculate a path from the agents’ current position to the destination point in the world. Next, the agent will move along the path until it reaches it destination.
This system is then reused for the player. When the player clicks on the screen, the Pointer feature in our game will then retrieve the point in the world that was clicked on and parse it to the Pathfinding system as the destination point for the player agent.

Conditional Pathfinding

Whilst path following, agents will contentiously check numerous conditions and react accordingly.
One such condition is to check for obstacles that were added on their path since they last calculated a path from their current position to their destination.
If an obstacle is encountered, the pathfinder will then attempt to find a new path to their destination, if none can be found, the agent will move to the next best position closest their destination.

Technical Design

Grid System

In order for our agents to navigate between points in the world, they need a system which tells them where they can and cannot walk.
We also need this system to be adjustable, flexible and optimised seeing as we regularly update the environment when placing new structures or remove objects like resources.
Therefore, we have opted for using a grid-based node system which will determine the walkability of surfaces and node connections based on various, easily expandable conditions and attributes.

Grid Creation

Various attributes determine how the grid is created and under what conditions nodes are set to walkable. Most of these attributes are editable in the inspector or are influenced by other systems.
Grid Attributes - Include, but are not limited to:
Maximum height of the grid
Determines how high the grid continues to detect walkable surfaces.
Grid Size
Determines the bounds of the grid in the world. Surfaces outside of these bounds are not used.
Cell Size
Determines the size of the cells which divide the grid.
Maximum height difference between connections
Determines how high nodes can be away from each other before the grid marks the connection as unwalkable.
Maximum connection distance
Determines how far away nodes can be away from each other before the connection is marked as unwalkable.
Layers
Walkable: Layers used for detecting walkable surfaces in the grid.
Obstacle: Layers used for detecting obstacles in the grid..

Considerations on Elevation Levels

Due to the nature and simplicity of the game, we didn’t feel it was necessary to account for multiple levels of elevation in our grid, such as tunnels, which means that the walkability of nodes is determined from a top-down perspective.

Cells

To account for larger grids with thousands of nodes, we divide the grid into cells which can be individually updated for increased performance. Nodes are added to their respective cells by comparing the position of a node to the position and size of a cell.
When a cell is updated, each neighbouring cell is automatically also updated, this prevents inaccuracies between nodes when updating near the edges of a cell.

Nodes

Nodes are the fundamental elements that form the basis for pathfinding. They each hold data about their exact location in the world, their walkability status, and their connected neighbouring nodes. This data allows the pathfinding algorithm to calculate the optimal path for the agent.

Pathfinding

Now that we have a surface for agents to find a path on, we need an algorithm which calculates these paths. Seeing as we are using a grid and to save time, we only consider the A* & Dijkstra algorithms, as both are well known ways for pathfinding on a grid in video games.
A* vs Dijkstra

A* Pros

Efficiency: A* can quickly find the shortest path to the target, optimizing the unit's travel time and allowing for more responsive gameplay.
Heuristic Function: The heuristic can be designed to consider different terrain costs and threats, guiding units through safer or faster routes, depending on game strategy.
Dynamic Adaptation: A* can efficiently recalculate paths if the environment changes (e.g., an obstacle appears), ensuring units adapt to evolving game conditions.
Scenario: A real-time strategy (RTS) game like StarCraft or Age of Empires.

Dijkstra Pros

Whole Map Analysis: When new objects are added, Dijkstra’s can help calculate the shortest path from these objects to all parts of the grid, aiding in effective placement.
Path Distribution: It can be used to simulate traffic flow by calculating potential routes for all agents in the system, helping in the design of road networks that minimize congestion.
Service Optimization: Ensures that the service coverage (like garbage collection routes or bus lines) reaches all necessary parts of the city efficiently.
Scenario: A city-building or simulation game like SimCity or Cities: Skylines.

Conclusion

Seeing as the pros of A* better fit the purposes for which we want to use pathfinding (moving objects between two specific points efficiently) and the example scenarios fit our Concept Inspirations, we will use the A* Algorithm in our game.

Finding a Path

The Pathfinder of the agent begins by converting the current position of the agent and its destination into nodes on the grid. This is done using the GetNodeFromWorldPosition  and GetClosestWalkableNode methods of the GridSystem class.
ALT

Open/Closed Sets

Next, the Pathfinder of the agent initialises the open and closed sets. The open set contains nodes that need to be evaluated, while the closed set contains nodes that have already been evaluated. The start node is added to the open set as the initial node to be evaluated.
The Pathfinder then enters a loop that continues as long as there are nodes in the open set. Inside the loop, the Pathfinder first calls the GetNodeWithLowestCost method to get the node in the open set with the lowest total cost (the sum of the goal cost and the heuristic cost). If no such node exists, the Pathfinder breaks out of the loop.
The Pathfinder then moves the current node from the open set to the closed set using the MoveToClosedSet method. If the current node is the target node, the Pathfinder has found a path, and it calls the RetracePath method to construct this path from the start node to the target node.
If the current node is not the target node, the Pathfinder calls the ProcessNeighbours method to process all the neighbours of the current node. This method updates the costs of the neighbours and adds them to the open set if they are not already in it.

Waypoint Followers

An extension on Path Followers, these Waypoint Followers function the same as Path Followers. However, instead of creating a path directly to their final destination, they will create paths in between every subsequent waypoint in their list and then generate a complete path based on that.
ALT

Obstacles

For performance as well as immersion, paths are not updated continuously whilst the agents is following the path. Instead, it performs a Raycast check outward to the front of its body which checks for any kind of obstacles which may have appeared along its path since it was last updated or created.

Reacting to Obstacles

When encountering an obstacle agents will, by default, react by trying to find a new path to their destination. If none can be found, the agent will stop until given a new destination and/or path.
Waypoint Followers
For Waypoint Followers, the agent will attempt to construct a new path towards their current waypoint and continue from there.
Additionally, the agent will also create a hypothetical path directly to the next waypoint to see if this path is faster than it is to first go to their current waypoint. This is to prevent Waypoint Followers from taking inefficient paths when an obstacle is right in front of their current waypoint.
Lastly, if a Waypoint Follower is unable to generate a path due to obstacles, they will wait until the obstacle they cannot get past has been removed before continuing their path.
ALT

Networking

In accordance with the Networking Workflow, the grid is handled by the server and pathfinding is requested by the client agent to the server, who then finds the path and sends it back to the player. This is to prevent client-side influence on pathfinding which could cause issues whilst synchronising with other players and the server, it also prevents players from being able to cheat through objects.

Pathfinding Client/Server Flow

Client request → Server creates & verifies path → Broadcast path back to Client.

Replication

We have considered adding replication to Pathfinding and Movement as this would smoothen the experience on the Client. However, due to time restrictions we will only add this if we have enough time for it.
