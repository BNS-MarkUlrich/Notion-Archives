Description

This design document outlines our multiplayer networking architecture for an HTML5 game. It details our rationale for a server-authoritative model, the selection of FishNet as our networking solution, and the considerations involved in transitioning from local to online multiplayer with a focus on prototyping and cost-effectiveness for short gameplay sessions.

Functional Design

Our objective is to optimize the player joining experience in HTML5 games by seamlessly connecting players to active servers. HTML5 games are often designed for quick engagement and short session lengths which also aligns with the core pillars of our client, Linx Interactive, so we've opted to eliminate manual lobby selection and server connections. This ensures players can jump into gameplay with minimal delay, aligning with the expectation of instant accessibility that's typical of HTML5 games.

Technical Design

Choosing Multiplayer Networking Solutions

We carefully evaluated several multiplayer networking solutions to facilitate server communication in our game. Our primary focus was on Photon and FishNet, as these are the most well known networking solutions for unity and we also had some experience with both. To make an informed decision, we weighed their respective strengths and weaknesses:
Unity Photon v.s. FishNet
Unity Photon
Pros:
Deep Unity Integration: Designed for seamless operation within the Unity ecosystem.
Ease of Use: Prioritizes a straightforward setup process and a gentler learning curve.
Cons:
Limited Features (Currently): As it's still under development, its feature set is less extensive than more mature solutions.
Uncertainty: Questions may remain about its long-term roadmap and dedicated support.
Lack of Knowledge: We have almost no prior knowledge of Photon.
Performance: Photon is a rather heavy solution which is not optimal for a small HTML5 game.
FishNet
Pros:
Open-Source: Flexibility for customization and potential for community contributions.
Mature and Feature-Rich: Well-established, offering a comprehensive range of networking features.
Prior Knowledge: We already have experience using FishNet
Performance: Fishnet is a rather light weight solution which is great for a small HTML5 game.
Web socket: FishNet also provides a great web socket api for porting the game to web.
Active Community: Provides access to support and resources.
Cons:
Steeper Learning Curve: May require greater technical expertise to set up and utilize compared to Photon.
Reliance on Community: Support primarily depends on community engagement, potentially less streamlined than official channels.
Decision:
We ultimately chose FishNet due to our existing familiarity with the solution, its robust feature set, and its generally lightweight nature, which is advantageous for HTML5 games running in a browser. FishNet's web socket API is crucial for enabling multiplayer networking within the constraints of the HTML5 environment.

Implementing FishNet

Design Principle: Server Authority

We adopted a server-authoritative model to enhance our game's security and reduce the potential for cheating. In this architecture, the server maintains ultimate control over game calculations and state updates. This approach minimizes the impact of client-side tampering on the overall game experience.

Local Multiplayer Setup

We initiated our development with local multiplayer for straightforward setup and testing. Here are the key FishNet components we integrated:
Network Manager: The foundation for client and server operations, bridging configuration and networking components.
Observer Manager: Controls object visibility across clients, streamlining player and object tracking within the server.
Player Spawner: Manages player object instantiation upon client connection, ensuring a seamless onboarding experience.
Tugboat: The backbone of FishNet communication, enabling reliable message exchange between game instances.
Time Manager: Synchronizes timing-related events and processes across clients with varying network speeds, promoting fairness.
Network Object: This component empowers any GameObject with network capabilities. It orchestrates automatic state synchronization (e.g., position, rotation) and ownership management for consistency across the network.

Code Integration

We strategically employed these methods for client-server communication:
ServerRPC: Marks methods for client-to-server input transmission, allowing the server to process actions securely.
Server: Denotes server-exclusive methods for managing authoritative game logic, inaccessible to client manipulation.
ObserverRPC: Facilitates the broadcast of state changes from the server to all clients, ensuring a synchronized view of the game world.

Online Multiplayer

Server Hosting
We selected PlayFlow as our server hosting solution due to its cost-effectiveness (free plan) and seamless Unity integration, making it well-suited for our prototyping needs. PlayFlow's 1-hour server session limit aligns with the anticipated short session lengths for testing our HTML5 game.
Network Configuration Manager
This custom component streamlines client/server transitions. When built as a server, it facilitates the conversion of a client instance into a server client, and conversely, when built as a user client, it automatically triggers connection to the designated server address. This adaptability simplifies the player's experience.
Network Interface Console
This in-build console enhances our testing capabilities and network flexibility. It empowers us with the following commands:
server 1/0: Toggles the current client instance between a locally hosted server and a regular client, simplifying local testing.
connect 1: Initiates a connection to the predefined server address, providing a direct control mechanism.
connectip string: Allows connection to a specific IP address, enabling versatile network configurations.
